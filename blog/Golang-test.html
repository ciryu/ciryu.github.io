Golang测试总结

发表于 2019-12-10 更新于 2019-12-14 分类于 编码 ， Golang 阅读次数：
本文字数： 15k 阅读时长 ≈ 14 分钟

摘 要：

本手册详细介绍了Go语言单元测试方法，包括功能测试、基准测试用例的编写和执行，内存分配、CPU占用、协程阻塞等统计信息的查看，以及如何进行并发测试、设置常用测试选项以及不同场景下的测试方法等，并介绍了几种常用的第三方测试框架。

背景

1、函数、方法实现代码比较复杂，函数依赖比较多，层层调用。
2、函数需要环境和配置的支持，如获取linux环境的文件路径，解析配置文件、访问数据库以及http请求等，都需预设测试条件。
3、函数涉及到编码解码、加解密、认证等安全保护，这部分mock比较困难。
4、集成测试，涉及多个组件的交互

Go内置单元测试框架Testing

命名约定

测试代码位于以xxx_test.go结尾的源文件中，一般与源码在同一个package中。位于同一个package中的主要原因是：测试可以访问package中不可导出的变量，方法等元素。也可单独创建一个测试package，以package xxx_test命名，可import依赖包，用于涉及多个组件交互的集成测试，可以更灵活编写测试用例：

package bytes_test
import (
"bytes"
"io"
...
)

测试函数命名规则：
•以Test开头的功能测试函数：func TestXxx(*testing.T)
•以Benchmark开头的性能测试函数： func BenchmarkXxx(b *testing.B)
•以Example开头的样例代码

func ExampleHello() {
hl := hello()
fmt.Println(hl)
// Output: hello.
}

示例函数无需接收参数，但需要使用注释的Output:标记说明示例函数的输出值，未指定Output:标记或输出值为空的示例函数不会被执行。go doc工具会解析示例函数的函数体作为对应包/函数/类型/类型方法的用法。
示例函数需要归属于某个包/函数/类型/类型的方法，具体命名规则如下：
func Example() { … } // 包的示例函数
func ExampleF() { … } // 函数F的示例函数
func ExampleT() { … } // 类型T的示例函数
func ExampleT_M() { … } // 类型T的M方法的示例函数

// 多示例函数 需要跟下划线加小写字母开头的后缀
func Example_suffix() { … }
func ExampleF_suffix() { … }
func ExampleT_suffix() { … }
func ExampleT_M_suffix() { … }

测试结果可以用t.fatal，t.Error，t.Fail等来标注，需要注意的是，t.fatal会立即中断后续测试代码的执行，t.error则会继续执行后续的测试代码。在实际应用中，一个小的测试错误不应该终止整个测试，因此推荐使用t.error:

// Error is equivalent to Log followed by Fail.
func (c *common) Error(args ...interface{}) {
c.log(fmt.Sprintln(args...))
c.Fail()
}

// Errorf is equivalent to Logf followed by Fail.
func (c *common) Errorf(format string, args ...interface{}) {
c.log(fmt.Sprintf(format, args...))
c.Fail() //继续执行后续测试代码
}

// Fatal is equivalent to Log followed by FailNow.
func (c *common) Fatal(args ...interface{}) {
c.log(fmt.Sprintln(args...))
c.FailNow() //立即返回
}

// Fatalf is equivalent to Logf followed by FailNow.
func (c *common) Fatalf(format string, args ...interface{}) {
c.log(fmt.Sprintf(format, args...))
c.FailNow()
}

测试示例

被测试代码为demo.go，相应的测试代码为demo_test.go，均在demo包下
demo.go文件内容如下：

package demo
import (
"fmt"
) 
func multiply(a, b int) int {
time.Sleep(time.Millisecond)
return a * b
}
func divide(a, b int) (int, error) {
if b == 0 {
return 0, fmt.Errorf("divide by zero")
} else {
 time.Sleep(time.Millisecond)
return a / b, nil
}
}
// 该函数未被测试到
func untest() {
fmt.Println("I am not tested")
}

demo_test.go文件内容如下：

package demo // 与被测文件放在同一个包中，方便测试非导出函数
import (
"testing" // 导入测试包
) 
// 功能测试函数名为TestXxx，Test之后的第一个字符必须是大写字母
// 入参类型必须为*testing.T
func TestMultiply(t *testing.T) {
t.Log("Test Multiply") // -v选项才输出
a, b := 2, 3
result := multiply(a, b)
if result != 6 {
t.FailNow() // 失败并停止执行当前测试函数
}
}
// 性能测试函数名为BenchmarkXxx，Benchmark之后的第一个字符必须是大写字母
// 入参类型必须为*testing.B
func BenchmarkMultiply(b *testing.B) {
// 使用b.N作为循环次数，测试调度器会自动计算循环次数
// 循环次数 * 函数平均执行时间 = benchtime
for i := 0; i < b.N; i++ {
multiply(10, 3)
}
} 
func TestDivideNormal(t *testing.T) {
a, b := 4, 2
result, err := divide(a, b)
if err != nil {
t.Error("should not return error") // 记录失败，继续执行当前测试函数
}
if result != 2 {
 t.Errorf("result should be 2, not %d", result)
}  
} 
func TestDivideZero(t *testing.T) {
a, b := 2, 0
_, err := divide(a, b)
if (err == nil) { // 无需进行后续检查
t.Fatal("should return error") // 记录失败，停止执行当前测试函数
}
if err.Error() != "divide by zero" {
t.Error("error(divide by zero) expected, not", err)
}
}

func BenchmarkDivide(b *testing.B) {
for i := 0; i < b.N; i++ {
divide(10, 3)
}
}

注：

  1. Go语言自带功能测试、性能测试、覆盖率测试框架。
  2. 测试代码必须保存在以*_test.go命名的测试文件中，执行go build编译时，不会编译测试文件。
  3. 功能测试函数命名符合TestXxx格式，Xxx为测试用例名，其第一个字母必须大写。
  4. 性能测试函数命名符合BenchmarkXxx格式，Xxx为测试用例名，其第一个字母必须大写。
  5. 性能测试使用入参的成员N（对应示例代码b.N）作为循环测试，调度器会根据设置的benchtime值自动计算循环次数。
    运行go test进行测试：

    go test

默认不会进行性能测试，要进行性能测试，需要增加-bench选项：

go test -bench .

注：.匹配任何字符，”-bench .”执行所有性能测试。
默认不会输出测试详细信息（用例执行、日志输出等）如果需要输出详细信息，需要增加-v选项：

go test -v

执行”Divide”相关的功能测试和性能测试：

go test -v -run Divide -bench Divide

测试命令使用

测试命令：go test [-c] [-i] [build and test flags] [packages] [flags for test binary]
相关帮助可运行go help test查看。

-c 仅编译，不执行测试
-i 安装测试包依赖的package，但是不运行测试
-bench regexp 执行正则表达式对应的性能测试用例，默认不执行性能测试用例
-benchmem 打印性能测试的内存分配统计信息
-benchtime t 性能测试运行的时间，缺省1秒，调度器会运行足够多次循环以达到这个时间 -benchtime 2s
-benchtime 10ms
-blockprofile block.out 在所有测试完成时输出协程阻塞统计文件
-blockprofilerate n 最小阻塞时间（纳秒），超过这个时间的阻塞事件将被记录，默认为1，即任何阻塞事件都记录
-cover 开启覆盖率分析
-covermode set|count|atomic 设置覆盖率分析模式
set：语句是否运行
count：语句运行次数
atomic：同count，但保证多线程安全
-coverpkg pkg1,pkg2,pkg3 对指定的包进行覆盖率分析，默认只对测试的包进行覆盖率分析（对测试包依赖的包不进行分析）
-coverprofile cover.out 输出测试覆盖统计分析文件
-cpu 1,2,4 程序运行使用的cpu个数（系统线程数）列表，默认为GOMAXPROCS
-cpuprofile cpu.out 输出cpu使用统计文件
-memprofile mem.out 输出内存分配用统计文件
-memprofilerate n 最小分配大小（字节），超过这个大小的内存分配将被记录，默认为512 * 1024，如果设置成1，任何内存分配都会记录。
-outputdir directory 设置输出目录，默认为go test执行所在目录
-parallel n 并发执行调用了t.Parallel的测试函数，n为同时执行测试函数的最大个数，默认值为GOMAXPROCS
-run regexp 运行正则表达式匹配的功能测试
-short 通知耗时测试缩短运行时间
-timeout t 测试执行的超时时间，超过此时间则panic -timeout 2m30s
-v 输出测试运行的详细信息，即使成功也会输出相关信息

注：
1.关于build flags，调用go help build，这些是编译运行过程中需要使用到的参数，一般设置为空。
2.关于packages，调用go help packages，这些是关于包的管理，一般设置为空。
3.关于flags for test binary，调用go help testflag查看，这些是go test过程中经常使用到的参数，如上表格从-bench regexp开始的选项都是flags for test binary。

注：
1.带f后缀的函数与不带f后缀函数的功能相同，只是增加了格式化输出字符串。
2.testing.T和testing.B均有如上这些函数。

测试类型

功能测试

功能测试函数以*testing.T类型为单一参数t，testing.T类型用来管理测试状态和格式化测试日志。测试日志在测试执行过程中积累，完成后输出到标准错误输出。
示例：

    }
}

基准测试

性能测试函数以接收*testing.B类型为单一参数b，性能测试函数中需要循环b.N次调用被测函数。testing.B类型用来管理测试时间和迭代运行次数，也支持和testing.T相同的方式管理测试状态和格式化测试日志，不一样的是testing.B的日志总是会输出。
启用内存使用分析，使用b.ReportAllocs()

func BenchmarkWriterFlush(b *testing.B) {
b.ReportAllocs()
bw := NewWriter(ioutil.Discard)
str := strings.Repeat("x", 50)
for i := 0; i < b.N; i++ {
bw.WriteString(str)
bw.Flush()
}
}

停止/重置/启动时间计值，使用b.StopTimer()、b.ResetTimer()、b.StartTimer()

func BenchmarkScanInts(b *testing.B) {
b.ResetTimer()
ints := makeInts(intCount)
var r RecursiveInt
for i := b.N - 1; i >= 0; i-- {
buf := bytes.NewBuffer(ints)
b.StartTimer()
scanInts(&r, buf)
b.StopTimer()
}
}

记录在一个操作中处理的字节数，使用b.SetBytes()

func BenchmarkFields(b *testing.B) {
b.SetBytes(int64(len(fieldsInput)))
for i := 0; i < b.N; i++ {
Fields(fieldsInput)
}
}

并发执行被测对象，使用b.RunParallel()和*testing.PB类型的Next()

func BenchmarkValueRead(b *testing.B) {
var v Value
v.Store(new(int))
b.RunParallel(func(pb *testing.PB) {
 for pb.Next() {
 x := v.Load().(*int)
 if *x != 0 {
  b.Fatalf("wrong value: got %v, want 0", *x)
 }
}
})
}

测试项

测试覆盖率

输出覆盖率信息，增加-cover选项：

go test -cover

观察覆盖率详细信息，可以增加-coverprofile选型输出覆盖率文件，然后使用go tool cover输出覆盖率详细信息：

go test -coverprofile cover.out

如上-func选项输出函数覆盖率统计信息。也可通过-html按html格式输出：

go tool cover -html cover.out

注：
1.未覆盖到的代码以红色字体输出。
2.将鼠标移到代码块，可以看到具体的执行次数。

// 并发执行性能测试
func BenchmarkParallel1(b *testing.B) {// 设置了并发协程数
b.SetParallelism(4) // 设置并发协程数为GOMAXPROCS的x倍
b.RunParallel(func(pb *testing.PB) { // 创建多个协程并发执行
for pb.Next() { // 调度器会将迭代分配到相应的协程中执行
 time.Sleep(time.Millisecond * 100)
}
})
}
func BenchmarkParallel2(b *testing.B) { // 未设置并发协程数
b.RunParallel(func(pb *testing.PB) {
for pb.Next() {
 time.Sleep(time.Millisecond * 100)
}
})
}
func BenchmarkParallel3(b *testing.B) { // 非并发执行
for i := 0; i < b.N; i++ {
 time.Sleep(time.Millisecond * 100)
}
}

运行测试：

go test -bench Parallel
go test -bench Parallel -cpu 4

注：
1.在单CPU时，未调用SetParallelism的与普通性能测试运行测试一样；设置-cpu 4之后，运行次数相对增多。
2.调用了SetParallelism运行次数比未调用的多。

白盒测试

白盒测试有访问包内部函数和数据结构的权限，因此可以做到一些普通客户端无法实现的测试。
下面的代码演示了为用户提供网络存储的web服务中的配额检测逻辑。当用户使用了超过90%的存储配额之后将发送提醒邮件。(译注：一般在实现业务机器监控，包括磁盘、cpu、网络等的时候，需要类似的到达阈值=>触发报警的逻辑，所以是很实用的案例)

gopl.io/ch11/storage1

package storage

import (
"fmt"
"log"
"net/smtp"
)

func bytesInUse(username string) int64 { return 0 /* ... */ }

// Email sender configuration.
// NOTE: never put passwords in source code!

这里有一个问题：当测试函数返回后，CheckQuota将不能正常工作，因为notifyUsers依然使用的是测试函数的伪发送邮件函数（当更新全局对象的时候总会有这种风险）。 我们必须修改测试代码恢复notifyUsers原先的状态以便后续其他的测试没有影响，要确保所有的执行路径后都能恢复，包括测试失败或panic异常的情形。在这种情况下，我们建议使用defer语句来延后执行处理恢复的代码。

func TestCheckQuotaNotifiesUser(t *testing.T) {
// Save and restore original notifyUser.
saved := notifyUser
defer func() { notifyUser = saved }()

// Install the test's fake notifyUser.
var notifiedUser, notifiedMsg string
notifyUser = func(user, msg string) {
    notifiedUser, notifiedMsg = user, msg
}
// ...rest of test...
}

这种处理模式可以用来暂时保存和恢复所有的全局变量，包括命令行标志参数、调试选项和优化参数；安装和移除导致生产代码产生一些调试信息的钩子函数；还有有些诱导生产代码进入某些重要状态的改变，比如超时、错误，甚至是一些刻意制造的并发行为等因素。

以这种方式使用全局变量是安全的，因为go test命令并不会同时并发地执行多个测试。
归纳：在编码时，应遵循一个函数一个功能的原则，将不同功能代码通过一个函数实现。上述测试代码中，为了便于测试，将发送邮件的代码放在一个单独的函数中，然后定义了一个函数变量，方便测试时替换发送邮件函数。

外部测试包

考虑下这两个包：net/url包，提供了URL解析的功能；net/http包，提供了web服务和HTTP客户端的功能。如我们所料，上层的net/http包依赖下层的net/url包。然后，net/url包中的一个测试是演示不同URL和HTTP客户端的交互行为，这样的行为在net/url包的测试代码中会导致包的循环依赖。

我们可以通过外部测试包的方式解决循环依赖的问题，也就是在net/url包所在的目录声明一个独立的url_test测试包。其中包名的_test后缀告诉go test工具它应该建立一个额外的包来运行测试。我们将这个外部测试包的导入路径视作是net/url_test会更容易理解，但实际上它并不能被其他任何包导入。

因为外部测试包是一个独立的包，所以能够导入那些依赖待测代码本身的其他辅助包；包内的测试代码就无法做到这点。在设计层面，外部测试包是在所有它依赖的包的上层。

通过避免循环的导入依赖，外部测试包可以更灵活地编写测试，特别是集成测试（需要测试多个组件之间的交互），可以像普通应用程序那样自由地导入其他包。

我们可以用go list命令查看包对应目录中哪些Go源文件是产品代码，哪些是包内测试，还有哪些是外部测试包。

Golang测试使用Testing框架能满足大部分测试场景，但是存在样板式代码较多，结合第三方框架封装，可以避免写太多样板代码，推荐Testify框架。
