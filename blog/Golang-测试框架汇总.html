第三方测试框架

goconvey单元测试框架

goconvey是一个BDD测试框架，对应的概念是行为驱动开发，可以跟go test集成。

BDD使用几乎近于自然语言的方式描述了软件的行为过程，因此，可以直接作为软件的需求文档，也可以直接应用到测试中，作为测试的标准文档。我们在做单元测试 时，经常是针对某个函数，或是某个类进行测试，但是被测函数或是被测的类是可能经常变化的，我们的测试案例也需要经常性的随之变化。

BDD描述的是软件的整个系统行为，几近于需求文档，可变性大大减小。因此，测试案例不需要做太大变化。同时，这样的测试案例最贴近于需求，贴近于实际的系统行为。

BDD描述的行为就像一个个的故事(Story)，系统业务专家、开发者、测试人员一起合作，分析软件的需求，然后将这些需求写成一个个的故事。开发者负责填充这些故事的内容，测试者负责检验这些故事的结果。通常，会使用一个故事的模板来对故事进行描述：

有了这样的故事、场景的描述，测试者可以通过一些BDD的测试框架将上面的故事转成测试代码（当然，也可直接由开发来完成，这里说测试者是为了方便理解），开发者实现产品代码，并保证测试代码通过。

测试实例

实例一

package convey1

import (
"testing"

. "github.com/smartystreets/goconvey/convey"
)

func TestSomething(t *testing.T) {
Convey("1 should equal 1", t, func() {
So(1, ShouldEqual, 1)
})

Convey("Comparing two variables", t, func() {
myVar := "Hello, world!"

Convey(`"Asdf" should NOT equal "qwerty"`, func() {
 So("Asdf", ShouldNotEqual, "qwerty")
})

Convey("myVar should not be nil", func() {
 So(myVar, ShouldNotBeNil)
})
})

Convey("This isn't yet implemented", t, nil)
}

Gocheck

测试实例

gocheck当然也支持基本测试、benchmark测试以及coverage测试，下面来看一个不是很复杂的例子（Assert和Check方法参数一样，区别就是Assert遇到错误会终止所有测试，而Check会继续执行，关于C的所有方法列表可以访问https://godoc.org/gopkg.in/check.v1#C)：

package hello_test

import (
"testing"
"io"

. "github.com/pingcap/check"
)

// Hook up gocheck into the "go test" runner.
func Test(t *testing.T) { TestingT(t) }
type MyTestSuite struct{}
var _ = Suite(&MyTestSuite{})
func (s *MyTestSuite) TestHelloWorld(c *C) {
c.Assert(42, Equals, "42")
c.Assert(io.ErrClosedPipe, ErrorMatches, "io: .*on closed pipe")
c.Check(42, Equals, 42)
}
func Fib(n int) int {
if n < 2 {
return n
}
return Fib(n-1) + Fib(n-2)
}
func benchmarkFib(i int, c *C) {
for n := 0; n < c.N; n++ {
Fib(i)
}
}
func (s *MyTestSuite) BenchmarkFib1(c *C)  { benchmarkFib(1, c) }
func (s *MyTestSuite) BenchmarkFib2(c *C)  { benchmarkFib(2, c) }
func (s *MyTestSuite) BenchmarkFib3(c *C)  { benchmarkFib(3, c) }
func (s *MyTestSuite) BenchmarkFib10(c *C) { benchmarkFib(10, c) }
func (s *MyTestSuite) BenchmarkFib20(c *C) { benchmarkFib(20, c) }
func (s *MyTestSuite) BenchmarkFib40(c *C) { benchmarkFib(40, c) }

运行一下基本测试（-check.v以及-check.vv参数可以看到更丰富的信息，-check.f可以指定测试用例):
$ go test

$ go test -check.v

$ go test -check.vv

$ go test -check.f MyTestSuite

$ go test -check.f “MyTestSuite.TestHelloWorld”

Ginkgo & Gomega

Ginkgo是Go语言行为驱动开发（BDD）类型的一个测试框架，Gomega是与Ginkgo相对应的匹配库。

测试实例

生成测试文件：

在为每一个package写测试用例之前，必须在该package下使用ginkgo bootstrap生成一个包名_suite_test.go的文件，例如，在package book下面使用ginkgo bootstrap生成了一个book_suite_test.go的文件，具体内容如下：

package books_test

import (
. "github.com/onsi/ginkgo"
. "github.com/onsi/gomega"

"testing"
)

func TestBooks(t *testing.T) {
RegisterFailHandler(Fail)
RunSpecs(t, "Books Suite")
}
  1. 为了将测试文件与源文件隔离，生成了与package book相对应的package book_test。
  2. 在该package下运行ginkgo或者go test命令将执行TestBooks函数。
  3. RegisterFailHandler是Ginkgo与Gomega的唯一连接，将Ginkgo的Fail函数注册到Gomega中，当测试结果不匹配时调用Fail函数通知Ginkgo该测试用例失败。
  4. RunSpecs函数告诉Ginkgo开始执行该package下的测试用例。

可以在book_suite_test.go文件中编写测试用例，但是通常都会将测试用例拆分到多个测试文件中，特别地，当一个package下面有多个文件时，可针对每个文件生成一个相应的测试文件。例如，package book下面有一个book.go的文件，使用ginkgo generate book生成一个相应的book_test.go测试文件，具体内容如下：

package books_test

import (
. "/path/to/books"
. "github.com/onsi/ginkgo"
. "github.com/onsi/gomega"
)

var _ = Describe("Book", func() {

})

生成了一个Describe的函数，Describe中的func将包含book.go文件中的所有测试用例，在其中添加一些测试用例，如下所示：

var _ = Describe("Book", func() { 
var (
    longBook  Book
    shortBook Book
)

BeforeEach(func() {
    longBook = Book{
        Title:  "Les Miserables",
        Author: "Victor Hugo",
        Pages:  1488,
    }

    shortBook = Book{
        Title:  "Fox In Socks",
        Author: "Dr. Seuss",
        Pages:  24,
    }
})

Describe("Categorizing book length", func() {
    Context("With more than 300 pages", func() {
        It("should be a novel", func() {
            Expect(longBook.CategoryByLength()).To(Equal("NOVEL"))
        })
    })

    Context("With fewer than 300 pages", func() {
        It("should be a short story", func() {
            Expect(shortBook.CategoryByLength()).To(Equal("SHORT STORY"))
        })
    })
})
})

组织测试用例：

  1. Describe用于描述某个行为

  2. Context用于描述某个行为在某种场景下

  3. It用于表述结果

  4. Describe下可嵌套Describe、Context和It，Context下可嵌套Context和It，Describe或者Context下的所有It的公共设置可提取到该Describe或者Context下的BeforeEach中处理，相应地，可在AfterEach中进行销毁。

  5. 在Describe或者Context中可声明变量，但只能在BeforeEach中初始化该变量，不可在Describe或者Context的block内初始化（一旦某个It修改了该变量，将会影响其后的It结果），而且，在Describe或者Context的block内除了变量声明外不能有额外的代码，所有的断言需放在BeforeEach、It和AfterEach中，例如，BeforeEach中的断言可用于确保在It的公共设置过程中没有出现错误。

  6. 正常情况下，当某个测试用例失败时，Ginkgo可以捕获并恢复异常，然后执行剩余的测试用例，但是当在It中创建一个新的goroutine时，若该goroutine中的断言失败，Gingko无法恢复panic，从而导致整个suite test panic，从而无法执行剩余的测试用例，可使用defer GinkgoRecover()避免该情形，具体如下所示：

  7. 当一个package内的所有测试用例存在全局公共的设置时，可在book_suite_test文件的BeforeSuite方法中设置，并在相应的AfterSuite方法中消除。注意，当BeforeSuite失败时，所有的测试用例都不会被执行，而当某个测试用例失败不会影响AfterSuite的执行，BeforeSuite和AfterSuite只能定义一次，当并行测试时，每个进程都会执行BeforeSuite和AfterSuite方法。

  8. 应该尽量保证It、BeforeEach等短小精悍，但是当进行集成测试时，会遇到复杂的工作流，此时可以使用by来组织测试用例，如下所示：

当测试用例通过时，只输出Ginkgo的绿点，不会输出by中的字符串，当测试用例失败时，会输出by中的字符串，到促发失败的地方。

运行测试用例

在每个package下面运行ginkgo 或者go test

（1）可在Describe、Context、It和Measure前加P或者X将其声明为Pending，其中的测试用例将不会被运行，但是会输出其中的string，可通过设置–noisyPendings=false来阻止其输出。此外，Pending的测试用例不会促发suite失败，可通过设置—failOnPending来促发suite失败。
（2）如果只想运行某些特定的测试用例，有两种方式：
1）在Describe、Context或者It前加F；
2）使用–focus=REGEXP和–skip=REGEXP标志，分别运行匹配和不匹配正则表达式的测试用例。注意：1）在构建CI时，当测试用例中含有focused的test时，Ginkgo退出的状态码是非0的，当指定了—focus或者—skip时，退出的状态码为0；2）当叶子结点被标记为focused，则其标记为focused的父结点都会被标记为unfocused，例如:

（3）默认情况下，Ginkgo会随机挑选测试文件运行，当运行某个测试文件中的测试用例时，则会按其在文件中的顺序执行，可通过设置—randomizeAllSpecs随机运行所有的测试用例。
（4）Ginkgo支持并行测试，将所有的测试用例划分到不同的go test 进程中，由于BDD的闭包共享context，所以并行的效果不是很好。命令：1）ginkgo –p，当runtime.NumCPU() <= 4时，使用runtime.NumCPU()个进程，当runtime.NumCPU() > 4时，使用runtime.NumCPU()-1个进程；2）ginkgo –nodes=N显示声明使用N个进程。

注意：
1）-p与–nodes无需同时声明；
2）并行测试的结果会被合并，若想实时查看每个进程的输出，可以通过设置-stream；
3）每个并行进程都会执行BeforeSuite和AfterSuite方法。

UT

Ginkgo不支持打桩，其作者认为打桩可以通过DI和interface来解决，但是提供与gomock的集成。
1、Gomock: 基于interface的mock（beego框架以及自研组件均未使用接口，若使用Gomock，源码需要很大改动，故未深入研究）
http://godoc.org/code.google.com/p/gomock/gomock
2、withmock: 基于package的mock（该方法适用于现有框架和组件，但是在安装和配置出现很多问题，需进一步探索）
http://godoc.org/github.com/qur/withmock
3、重定义函数：重新定义依赖包中的函数（目前采用该方法，需修改少许源码，待探索解决方法）。
http://blog.natefinch.com/2014/04/mocking-functions-in-go.html

Gotests

Gotests测试框架能够框架自动生成整个包或者文件的test文件，再向TODO添加具体测试用例，可免去许多重复代码的编写。

测试步骤

进入到需要生成test文件的目录，命令行执行 gotests -all -w file.go即可生成对应的file_test.go文件。后续只要在test函数里增加对应的不同参数的用例即可。
